三种架构方案对比

基于需求分析：侧边栏 + 双栏布局 + 5大新模块 + 现有视图迁移

核心决策：如何从「水平导航 SPA」演进到「侧边栏双栏布局」？

A

渐进式迁移

保留现有 App.tsx 主体结构，新增 Sidebar 组件作为可选覆盖层；通过配置开关逐步将功能从 Settings 中移出；过渡期同时存在新旧两种导航入口

Pros

  • 风险最低，每步可独立发布
  • 现有功能零影响
  • 可随时回滚

Cons

  • 过渡期 UX 混乱（双导航）
  • 技术债累积
  • 最终仍需大重构
B

全新 Shell + 组件复用

新建 AppShell 组件（Sidebar + ContentArea），现有视图组件（UnifiedMcpPanel/PromptPanel 等）封装为 Page 包装器放入新布局；新页面全新建造；Settings 功能按页面迁移

Pros

  • 架构清晰，一步到位
  • 复用 80% 现有组件
  • 不影响 Rust 后端

Cons

  • App.tsx 改动较大
  • Settings 需拆解迁移
  • 需完整回归测试
C

全量重写前端 Shell

抛弃现有 App.tsx，从头构建全新的布局系统、路由体系、页面组件；Settings、导航、视图全部重新设计实现

Pros

  • 最干净的代码
  • 无历史包袱

Cons

  • 耗时最长（4-8周）
  • 需重写大量已验证逻辑
  • 高风险，回归范围巨大
  • 与 Tauri 后端交互需重新对接
